# Core Principles & Decision Checklist

# Arduino IDE + ESP32/ESP8266 + Optional SquareLine Studio

> Every AI coding agent must read this file before writing any code.

---

## ASSUMPTIONS

- Primary targets are **ESP32** and **ESP8266** using the **Arduino language in Arduino IDE**.
- Production firmware may include Wi-Fi, sensors, relays, displays, local storage, OTA updates, and optional LVGL-based UI generated by **SquareLine Studio**.
- Default security level is **medium-high** for connected consumer or prosumer devices.
- Default deployment model is:
  - UART flashing during development
  - OTA for production field upgrades on ESP32
  - OTA optional and board-specific on ESP8266
- Default UI stack, when SquareLine is present, is **LVGL-based** and must be treated as generated code that is not manually edited unless explicitly required.
- Default preference is **performance and reliability over feature density**.

## Non-Negotiable Principles

**Write deterministic firmware.**  
No hidden side effects. No long blocking code in `loop()`. Time-sensitive work must be explicit, measurable, and bounded.

**Prefer explicit hardware boundaries.**  
All board pins, buses, device addresses, calibration values, and electrical assumptions live in dedicated board and hardware config files. Never scatter pin numbers across the codebase.

**Keep `loop()` cooperative and fast.**  
`loop()` must behave like a scheduler tick, not a work dump. Long delays, busy waits, and synchronous retries are forbidden in production code.

**Generated UI code is not business logic.**  
SquareLine or LVGL generated files are treated as generated artifacts. Application logic lives in wrappers, controllers, event bindings, and adapter layers outside generated files.

**Every I/O operation must fail safely.**  
Sensors disconnect. Wi-Fi drops. NTP fails. File systems corrupt. HTTP servers get partial requests. Every external dependency must have timeout, retry policy, and degraded-mode behavior.

**Memory discipline is mandatory.**  
Heap fragmentation, oversized buffers, dynamic `String` churn, and uncontrolled JSON parsing are not acceptable. Prefer fixed-size buffers, `constexpr`, flash-resident constants, and bounded allocations.

**Security defaults must be conservative.**  
Never hardcode secrets. Never expose unauthenticated debug or config endpoints in production. Do not ship OTA, web config, or MQTT control paths without authentication and an explicit authorization model.

**Observability is built in, not bolted on.**  
Every production firmware must expose enough logs, counters, health signals, and version metadata to diagnose field failures.

**Design for recovery, not perfection.**  
A device must recover from brownouts, Wi-Fi loss, bad config, watchdog resets, failed OTA, missing peripherals, and corrupted preferences with a defined fallback path.

**Repo scanning is mandatory before code generation.**  
Before writing or changing code, the AI agent must scan the repository structure. If SquareLine files exist, the agent must study them to infer:
- LVGL version
- Generated folder layout
- Screen names and widget IDs
- Event callback stubs
- Display/touch driver assumptions
- Asset packaging strategy
- Existing theme, font, and image usage

**Do not silently change generated artifacts.**  
If the repo contains generated SquareLine output, any regeneration risk must be called out. The AI agent must preserve custom code outside generated directories.

## Production Defaults

### Default architectural stance

- **MUST** support clean separation between:
  - board support
  - hardware drivers
  - services
  - application logic
  - UI integration
- **MUST** support safe boot with missing network.
- **MUST** support safe boot with missing optional peripherals.
- **MUST** expose firmware name, version, build date, git commit, and board profile.
- **MUST** define a watchdog strategy.
- **MUST** define persistence strategy and migration behavior.
- **MUST** define reset reason handling.

### Default performance stance

- `loop()` iteration target: **sub-5 ms average**, excluding explicitly scheduled slow tasks.
- UI refresh and event handling must remain responsive.
- No periodic task may monopolize CPU.
- Dynamic allocations in hot paths are forbidden unless justified.

### Default reliability stance

- Every retry must have:
  - a maximum attempt or cooldown rule
  - jitter or backoff when network-related
  - a state transition on failure
- All field-updatable devices must define:
  - rollback or recovery strategy
  - factory reset entry path
  - corrupted-config recovery path

## Definition of Done

Code is not done until all items below are true.

### Functional

- [ ] Feature works on target hardware.
- [ ] Feature works after cold boot and warm reboot.
- [ ] Feature behavior under no-network conditions is defined.
- [ ] Feature behavior under invalid config is defined.
- [ ] Feature behavior under missing peripheral conditions is defined.

### Code quality

- [ ] No copy-pasted pin literals outside config files.
- [ ] No `delay()` longer than a small startup wait unless explicitly justified.
- [ ] No blocking loops waiting forever on Wi-Fi, sensors, serial input, or HTTP responses.
- [ ] No business logic inside generated UI files.
- [ ] All module APIs have clear ownership and failure contracts.

### Performance

- [ ] No uncontrolled heap growth over runtime soak test.
- [ ] No excessive `String` concatenation in recurring paths.
- [ ] No unnecessary redraws for LVGL/SquareLine projects.
- [ ] Sensor polling intervals are explicit and justified.
- [ ] Network reconnect logic uses bounded retries.

### Security

- [ ] No secrets committed to repo.
- [ ] Debug endpoints gated or removed in production.
- [ ] OTA path authenticated if enabled.
- [ ] Config reset path requires explicit user intent.
- [ ] MQTT/HTTP command channels validate payloads and reject malformed input.

### Operations

- [ ] Firmware version is visible in serial logs and status endpoint or UI.
- [ ] Reset reason is logged on boot.
- [ ] Device health indicators are available.
- [ ] Factory reset behavior is documented.
- [ ] Flashing and provisioning steps are documented.

## Repo Scan Checklist

Before generating any firmware, the AI agent must inspect the repository and answer:

- [ ] Is this an ESP32 project, ESP8266 project, or multi-target project?
- [ ] Which board packages and libraries are already in use?
- [ ] Is there a `lib/`, `src/`, `include/`, or flat Arduino sketch structure?
- [ ] Are there generated UI folders such as `ui/`, `UI/`, `squareline/`, `generated/`, `screens/`, `assets/`, `fonts/`, `images/`?
- [ ] Are LVGL files present?
- [ ] Is SquareLine metadata or export output present?
- [ ] Which display and touch libraries are referenced?
- [ ] Is there existing OTA or Preferences/LittleFS code?
- [ ] Is there evidence of FreeRTOS tasks on ESP32?
- [ ] Are credentials or `.env`-style placeholders already defined?
- [ ] Are there board-specific pin maps or wiring docs?
- [ ] Are there migration or backward-compatibility constraints from prior releases?

If SquareLine or LVGL files exist, the AI agent must additionally inspect:

- [ ] `ui.h`, `ui.c`, `ui_events.*`, `ui_helpers.*`
- [ ] screen initialization functions
- [ ] widget identifiers and naming patterns
- [ ] custom fonts, images, theme files
- [ ] display buffer size assumptions
- [ ] asset storage strategy: PROGMEM, flash FS, SD card
- [ ] callback boundaries where business logic should attach

## Decision Checklist

Use this before implementing any feature.

### Hardware and board

- [ ] Does the feature require new pins, buses, interrupts, or power sequencing?
- [ ] Is the feature safe on both ESP32 and ESP8266 if multi-target?
- [ ] Does the feature need debouncing, filtering, or calibration?

### Scheduling

- [ ] Can the feature run from cooperative `loop()` scheduling?
- [ ] Does the feature require an interrupt?
- [ ] On ESP32, does the feature truly require a dedicated FreeRTOS task?

### Storage

- [ ] Is this data a config value, runtime state, log sample, or asset?
- [ ] Should it live in Preferences/NVS, LittleFS, SD, or RAM only?
- [ ] What happens if persisted data is missing or invalid?

### UI

- [ ] Is the UI generated by SquareLine?
- [ ] Are changes better made in SquareLine rather than by manual edits?
- [ ] Are custom event handlers isolated from generated files?

### Networking

- [ ] What happens if DNS fails, NTP fails, AP is unavailable, or broker is down?
- [ ] Is there a backoff strategy?
- [ ] Is offline mode acceptable and defined?

### Security

- [ ] Does this expose configuration or control surfaces?
- [ ] Is authentication required?
- [ ] Could malformed input crash the device or exhaust memory?

## Common Pitfalls

- Editing generated SquareLine files directly, then losing changes on re-export
- Using `String` heavily in loops and callbacks
- Blocking boot on Wi-Fi forever
- Mixing UI updates with networking or sensor reads in one monolithic module
- Ignoring heap/stack limits on display-heavy projects
- Shipping with verbose debug serial logs enabled
- Using default credentials in production
- Assuming ESP32 and ESP8266 APIs are interchangeable
- Overusing interrupts for logic that should be polled
- Writing to flash too frequently and reducing device lifetime

---

## Failure Modes

- Boot loop caused by invalid persisted config
- Watchdog reset due to long display or network operation
- LVGL redraw stalls due to oversized screen updates
- LittleFS mount failure after abrupt power loss
- Brownout resets during relay, backlight, radio, or motor activity
- OTA image too large for current partition layout
- Wi-Fi reconnect storm causing UI starvation
- Heap fragmentation from repeated JSON/string allocation
- Sensor bus lockup causing startup hang
- Touch/display mismatch after SquareLine export assumptions change
